# rw-ai v0.5 — Implementation Plan

> Companion to [`v0.5-tui-control-tower.md`](./v0.5-tui-control-tower.md). That
> document defines the architecture; this one breaks it into ordered, testable
> work units. Read that first.

## Conventions

- **Strict TDD**: every work unit is RED → GREEN. Write the failing test first.
- **Match the existing codebase**: `Result<T, E>` returns (no throwing across
  layers), injectable side effects (`now`, `exec`, watchers) for determinism,
  atomic file writes, zod at every boundary, agent-agnostic, secrets never on disk.
- **Slices ship independently**: each phase leaves the tool usable. No phase depends
  on a later one.
- **One writer thread** per work unit; reviews run with fresh context after each PR.

## Dependency order

```
WU-0 (hygiene) ─┐
                ├─► Phase 1 (read model) ─► Phase 2 (daemon) ─► Phase 3 (nvim)
                │                                                   │
                └───────────────────────────────────────────►  Phase 4 (diagrams)
                                                                    │
                                                          Phase 5 (external, opt.)
```

Phase 4 needs the review buffer from Phase 3. Phase 5 needs the daemon from Phase 2.

## Phase 1 — Read model + hook-marker contract

Goal: compute current workspace state from structured signals, no daemon. Ships as
`rw status`.

| WU | Task | Test (RED first) |
| --- | --- | --- |
| 1.1 | `src/state/marker.ts`: zod schema for a hook marker written to `.review-board/sessions/<id>/status.json` — `{ version, sessionId, phase: 'idle'|'working'|'blocked'|'review', detail?, tests?: 'passed'|'failed'|'unknown', updatedAt }`. Parse + validate returning `Result`. | Valid/invalid/missing marker cases. |
| 1.2 | `src/state/git.ts`: `readGitSignal(worktreePath, baseBranch, { exec })` → `{ dirty, staged, ahead, behind }`. Injectable `exec` like `detectGitBranch`. | Mock exec for clean/dirty/ahead-behind; failure → safe defaults. |
| 1.3 | `src/state/aggregate.ts`: `computeSessionState(config, claims, markers, gitSignals)` → `SessionState[]` with a derived `light: 'green'|'yellow'|'red'` from explicit rules (documented in-code). Pure function. | Truth-table test mapping (claim × marker × git) → light. |
| 1.4 | `rw status` command in `cli.ts`: prints human table + `--json`. One-shot read; no writes. | `cli.test.ts`: json shape, exit codes, empty-config case. |

**Light derivation rules (Phase 1 baseline, refine later):**
- `red`: marker `blocked` OR `tests: failed`.
- `yellow`: marker `working`/`review` OR git `dirty`/`ahead>0` with a held claim.
- `green`: claim free/idle and worktree clean.

**DoD**: `rw status` reports every session's state deterministically from disk;
fully unit-tested; no process left running.

## Phase 2 — Observer daemon

Goal: one lazy, per-repo, multi-client read/notify service over the Phase 1 model.

| WU | Task | Test |
| --- | --- | --- |
| 2.1 | Transport: per-repo pipe/socket address derived from the repo root (hash), like CodeGraph's `codegraph-<hash>`. Frame protocol (length-prefixed JSON). | Round-trip encode/decode; two addresses differ by repo path. |
| 2.2 | Lifecycle: single-instance guard (reuse the lock pattern), lazy start, **idle-timeout self-shutdown** (configurable, default 5 min). | Spawns once; second spawn attaches; shuts down after idle with no clients. |
| 2.3 | Watcher: watch `claims.json` + `sessions/*/status.json` + a cheap git-poll; recompute incrementally; skip unchanged (content-hash/mtime). | Change a marker → exactly one recompute; unchanged write → none. |
| 2.4 | Protocol: `subscribe` → initial snapshot + subsequent deltas; `get` → one snapshot; support N concurrent clients. | Two clients both receive a delta after one change. |
| 2.5 | `rw status` uses the daemon (spawn-if-absent), falling back to one-shot compute if the daemon can't start. | Daemon path and fallback path both green. |

**DoD**: multiple clients observe fresh state from one daemon; it dies when idle;
no always-on cost; `rw status` works with or without it.

## Phase 3 — Neovim plugin (STATUS + REVIEW)

Goal: the cockpit as a Neovim plugin. Native editing; no rebuilt editor.

| WU | Task | Test / verify |
| --- | --- | --- |
| 3.1 | Lua RPC client to the daemon (connect, subscribe, receive deltas). | Headless `nvim --headless` smoke: connect + first snapshot. |
| 3.2 | `:RwStatus` — sessions list with area, claim, git, light. | Snapshot assertion on rendered buffer lines. |
| 3.3 | `:RwReview <id>` — sessions pane + diff buffer (`git diff` vs integration branch) in a real editable buffer with the user's LSP/keymaps. | Buffer is `modifiable`; diff gutter present; `:w` writes the worktree file. |
| 3.4 | Evidence pane — read `.review-board/sessions/<id>/` (tests, summary, risks, handoff). | Missing evidence degrades to an empty pane, no error. |
| 3.5 | Actions: `approve`/`reject`/`comment` → write decision/risk record to backing store + release the `integrator` claim (via daemon or CLI). | Decision record persisted; claim transitions to free. |

**DoD**: from Neovim, review a session's diff, edit inline, see evidence, and record
a decision — all by keyboard. Does not run agents; does not merge.

## Phase 4 — Development diagrams from CodeGraph

Goal: consume the existing `.codegraph/` index; never build our own.

| WU | Task | Test / verify |
| --- | --- | --- |
| 4.1 | `src/codegraph/read.ts`: open the repo's `.codegraph/codegraph.db` read-only; query `edges` (calls/contains/imports) for a set of symbols. Absent DB → typed "unavailable". | Fixture DB: query returns expected edges; missing DB → graceful. |
| 4.2 | Blast-radius: changed files in the diff → changed symbols (via `nodes` file/line) → callers. | Given a diff + fixture graph, returns the caller set. |
| 4.3 | Render in the review diagram pane via `image.nvim` (Kitty/WezTerm/sixel); **ASCII fallback** when unsupported. | Fallback path renders text; supported path emits an image. |

**DoD**: the review table shows a call-graph / blast-radius for the diff, sourced
from CodeGraph, degrading to ASCII where the terminal can't render images.

## Risk register

| Risk | Mitigation |
| --- | --- |
| Terminal image support is uneven (only Kitty/WezTerm/sixel). | ASCII fallback is a first-class path, tested in Phase 4. |
| Daemon lifecycle races (double-spawn, stale socket). | Single-instance guard + idle-timeout; reuse the proven lock/nonce pattern. |
| Git polling cost on large repos. | Debounced watcher + incremental recompute; git status is cheap vs a full scan. |
| Scope creep back toward Option B (wrapping/embedding). | Boundaries in the architecture doc §5 are the review checklist for every PR. |

## Definition of done for v0.5

`rw status` from the CLI and Neovim shows every session's live state; the review
table lets the integrator review a diff, edit inline, see evidence and a CodeGraph
diagram, and record a decision — 100% keyboard, composing with Zellij/Tmux + Neovim,
with the headless brain unchanged as the single source of truth.
