# rw-ai v0.5 — Implementation Plan

> Companion to [`v0.5-tui-control-tower.md`](./v0.5-tui-control-tower.md). Aligned to
> `origin/main` (the shipping implementation), not a green-field scaffold. Read the
> architecture doc first, especially §4.1 (what already exists) and §4.3 (the
> daemon-vs-MCP constraint).

## Conventions

- **Strict TDD**: every work unit is RED → GREEN.
- **Match the existing codebase**: `Result`-style returns, injectable side effects
  (`now`, `exec`, watchers) for determinism, atomic writes, zod at boundaries,
  agent-agnostic, secrets never on disk. Reuse `src/engine/*` primitives.
- **Reuse before build**: origin already implements the contract, provisioning,
  claims lifecycle, MCP tools, hooks, and the integrator/review-board substrate. New
  code extends these; it does not re-implement them.

## Dependency order

```
Phase 0 audit ─► Phase 1 (read model) ─► Phase 2 (daemon) ─► Phase 3 (nvim)
                                                                 │
                                                       Phase 4 (diagrams)
                                                                 │
                                                       Phase 5 (external, reframed)
```

## Phase 1 — Read model + hook-marker contract

Goal: compute per-session state from structured signals; ship `rw status`. Reuse
existing engine primitives.

| WU | Task | Test (RED first) |
| --- | --- | --- |
| 1.1 | `src/state/marker.ts`: zod schema for `.review-board/sessions/<id>/status.json` — `{ version, sessionId, phase: 'idle'|'working'|'blocked'|'review', tests?: 'passed'|'failed'|'unknown', detail?, updatedAt }`. Parse/validate + atomic write helper (reuse `engine/fs-atomic.ts`). | Valid/invalid/missing marker. |
| 1.2 | **Wire the marker**: have `rw session-start` (`src/cli/commands/session-start.ts`) and/or a lifecycle wrapper write the marker on lifecycle transitions. Generalize the Claude-specific `engine/claude-sessions.ts` signal into the marker where available. | Hook writes a valid marker; agent-agnostic path covered. |
| 1.3 | `src/engine/git.ts`: add `aheadBehind(worktreePath, baseBranch, { exec })` → `{ ahead, behind }`. | Mock exec; detached/no-upstream → safe defaults. |
| 1.4 | `src/state/aggregate.ts`: `computeSessionState(config, claims, markers, gitSignals)` → `SessionState[]` with derived `light`. Pure function; reuse `identity.readClaims`, `git.statusPorcelain`/`changedFiles`. | Truth-table (claim × marker × git) → light. |
| 1.5 | `rw status` in `src/cli/run.ts`: human table + `--json`. One-shot read, no writes; add to USAGE. | `run` test: json shape, exit codes, empty-config. |

**Light rules (baseline):** `red` = marker `blocked` or `tests: failed`; `yellow` =
marker `working`/`review` or git dirty/ahead with a held claim; `green` = claim
free/idle and clean.

**DoD**: `rw status` (CLI) reports every session's state deterministically from
disk; markers are written by hooks; fully unit-tested.

## Phase 2 — Observer daemon (new; not the MCP server)

Goal: one lazy, per-repo, multi-client read/notify service over the Phase-1 model.
**The stdio per-agent MCP server cannot be reused as the daemon** (§4.3).

| WU | Task | Test |
| --- | --- | --- |
| 2.0 | **Transport decision — RESOLVED: Option A.** The daemon is a new read/notify service with its own pipe/socket transport and a length-prefixed JSON-frame protocol. `rw mcp` stays the per-agent stdio server for mutations (via `engine/identity.ts`) — unchanged this phase; any MCP→daemon read forwarding is optional and deferred. Rationale: the Phase-3 client is a simple Lua RPC (JSON frames are trivial in Lua, an MCP client is not); daemon = read/notify only, keeping claims single-owner; no MCP-SDK coupling; WU-2.1 already specifies JSON frames. | — (decision) |
| 2.1 | Transport: per-repo address from repo root (hash). **Cross-platform**: Windows named pipe (`\\.\pipe\rw-ai-<hash>`, primary target) vs Unix domain socket, behind one interface. Length-prefixed JSON frames. | Round-trip; addresses differ by repo; platform-selected. |
| 2.2 | Lifecycle: single-instance guard, lazy start, **idle-timeout self-shutdown**. | Spawns once; second attaches; idle shutdown. |
| 2.3 | Watcher: watch `claims.json` + `sessions/*/status.json`; cheap debounced git-poll; incremental recompute (skip unchanged via mtime/hash). | One change → one recompute; unchanged → none. |
| 2.4 | Protocol: `subscribe` → snapshot + deltas; `get` → snapshot; N concurrent clients. | Two clients both get a delta. |
| 2.5 | `rw status` prefers the daemon (`readSnapshotViaDaemon`), falls back to one-shot `collectSessionStates`. **Auto-spawn deferred**: `rw daemon` launches the daemon (foreground/detached by a launcher); the DoD "works with or without it" is met by the fallback, so detached auto-spawn from `rw status` (frail cross-platform, hard to test) waits for the Phase-3 launcher. | Daemon path + fallback both green. |

**DoD**: multiple clients observe fresh state from one daemon; dies when idle;
`rw status` works with or without it.

## Phase 3 — Neovim plugin (STATUS + REVIEW)

Goal: the cockpit as a Neovim plugin; native editing.

| WU | Task | Test / verify |
| --- | --- | --- |
| 3.1 | Lua RPC client to the daemon (connect, subscribe, deltas). | `nvim --headless` smoke: connect + first snapshot. |
| 3.2 | `:RwStatus` — sessions list (area, claim, git, light). | Snapshot of rendered buffer lines. |
| 3.3 | `:RwReview <id>` — sessions pane + editable diff buffer (`git diff` vs integration branch) with the user's LSP/keymaps. | Buffer `modifiable`; diff gutter; `:w` writes the file. |
| 3.4 | Evidence pane — read `.review-board/` (integrator report + new `sessions/<id>/`). | Missing evidence degrades gracefully. |
| 3.5 | Actions: approve/reject/comment → **new decision record** in the backing store + release the `integrator` claim via `engine/identity.ts`. | Decision persisted; claim → free. |

**Plugin location**: ship the Lua plugin in-repo under `nvim/` (or `rw-ai.nvim`
sibling), installable via the user's plugin manager. Decide before 3.1.

**DoD**: from Neovim, review a diff, edit inline, see evidence, record a decision —
by keyboard. Does not run agents; does not merge (`rw_finish`/`lifecycle.ts` stays
the deliberate merge path).

## Risk register

| Risk | Mitigation |
| --- | --- |
| Reusing the MCP server as the daemon (it is stdio/per-agent/single-client). | Build the daemon new; MCP stays an integration surface (§4.3, WU-2.0). |
| Terminal image support is uneven. | ASCII fallback is a first-class, tested path (Phase 4). |
| Daemon lifecycle races (double-spawn, stale socket). | Single-instance guard + idle-timeout; reuse the proven lock/nonce patterns in the engine. |
| CodeGraph internal schema versions and could break a direct DB read. | Query via its interface, never its SQLite file (WU-4.1). |
| Cross-platform transport (Windows named pipe vs Unix socket). | One transport abstraction, Windows-first (WU-2.1). |
| Scope creep back toward Option B (wrapping/embedding). | Boundaries (architecture doc §5) are the review checklist for every PR. |
| Daemon broadcast backpressure — a slow-reading subscriber grows Node's socket buffer. | Accepted for v0.5: snapshots are tiny (N sessions) and clients are the user's own editor/status; revisit with `write()` drain handling if payloads grow. |
| Windows named-pipe ACL not restricted (Node's `net` doesn't expose it) — a local user who knows the repo path could read non-secret state (branch/git/phase; the claim token is always redacted). | Unix socket is `chmod 0600`; Windows pipe hardening deferred. Documented gap, not a token leak. |

## Definition of done for v0.5

`rw status` from the CLI and Neovim shows every session's live state; the review
table lets the integrator review a diff, edit inline, see evidence and a CodeGraph
diagram, and record a decision — 100% keyboard, composing with Zellij/Tmux + Neovim,
built on the existing rw-ai core with claims still single-owned.
