# rw-ai v0.5 — Terminal Control Tower (aligned plan)

> Status: current direction, **aligned to `origin/main`** (the shipping
> implementation), not a green-field scaffold. rw-ai already implements the
> contract, provisioning, claims lifecycle, an MCP tool surface, agent hooks, and
> the integrator/review-board substrate. v0.5 adds the missing surface — a read
> model, an observer daemon, and a Neovim review cockpit — on top of that base.
>
> A parallel redesign line (`redesign-brain`, a smaller external-adoption core) is
> preserved as a branch; useful ideas from it (agent-agnostic identity, external
> worktree adoption, runtime manifests) are noted here but `origin/main` is the
> trunk.

## 1. Why this plan exists (the pivot)

The original intent of rw-ai: a **review table** plus **worktree and branch
administration** for parallel agent sessions across different **areas** of a
codebase — **feature-based** agent development, not layer-based.

Two intermediate UI directions were evaluated and rejected against the real flow:

| Tool | Verdict | Reason |
| --- | --- | --- |
| ORCA IDE | Rejected | Too slow, not keyboard-friendly, uncomfortable for a terminal-first flow. |
| Herdr | Rejected | Expected to be the solution; did not fit in practice. |

The decision: rw-ai provides its **own terminal-native surface** — 100%
keyboard-driven, fast — that **composes** with the tools the user already lives in
(**Zellij/Tmux + Neovim/LazyVim**) instead of replacing them.

## 2. Product identity — "Option A" (observe & compose)

rw-ai stays a **headless control-plane** and gains a **thin, read-oriented surface**
that **observes** structured state. It does **not** wrap agent terminals, manage
panes, or embed an editor.

Rejected alternative ("Option B": a daemon that PTY-wraps agents, parses stdout for
state, and embeds terminals in a full-screen TUI). It rebuilds what Zellij/Tmux do
better and reintroduces ORCA's latency. **Guiding rule: compose, don't compete.**

## 4. Components

### 4.1 Existing rw-ai core — already built on `origin/main` (reuse)

Contrary to a green-field assumption, most of the coordination layer exists:

- **Contract** (`src/contract/schema.ts`): agent-agnostic `agents.config.json` —
  project/stacks, git config, board dir, db (no secrets), `sharedZones`, `tests`,
  `sessions[]` (`id/branch/worktree/status/areas/ports/db/platforms`), claims
  (`free`/`claimed`, TTL), roles (`sessionId | integrator`).
- **Provisioning engine** (`src/engine/`): `worktrees.ts`, `branches.ts`,
  `database.ts`, `configure.ts`, `bootstrap.ts` create/wire real worktrees, branches,
  DBs and `.env.local`. This is first-class provisioning, not external adoption.
- **Claims lifecycle** (`src/engine/identity.ts`): a **single in-repo owner** for
  claim/release/read, exercised by both the CLI and the MCP tools.
- **MCP tool surface** (`src/mcp/server.ts` + `tools.ts`): 10 tools — `rw_status`,
  `rw_roles`, `rw_whoami`, `rw_claim`, `rw_release`, `rw_check`, `rw_finish`,
  `rw_lane_check`, `rw_bootstrap`, `rw_add_remote`. **Note:** the server is **stdio,
  per-agent, single-client** — an integration surface, not a shared daemon (§4.3).
- **Agent hooks + adapters** (`src/engine/adapters.ts`): writes `.mcp.json`,
  `.claude/settings.json`, `opencode.json`; installs `SessionStart → rw
  session-start` and `PreToolUse → rw lane-guard`.
- **Integrator + review-board substrate** (`src/engine/integrator.ts`, `board.ts`):
  cross-session integration analysis (invasions, overlaps, simulated merges) →
  `.review-board/reports/integration.md`; seeds `.review-board/` (screenshots,
  test-artifacts, reports, tasks, `board.md`).

This is the single source of truth. Every new surface reads from it.

### 4.2 Structured state signals (define + wire, then read)

Agent state (🟢 idle · 🟡 working/attention · 🔴 blocked/failed) is derived from
structured signals, never regex over terminal output. Sources:

1. **`claims.json`** — role occupancy (via `engine/identity.ts`). Exists.
2. **git state per worktree** — via `engine/git.ts` (`statusPorcelain`,
   `changedFiles`, `currentBranch`); **add ahead/behind** (missing today).
3. **hook markers** — a new agent-agnostic contract
   `.review-board/sessions/<id>/status.json` (`phase`, `tests`, `detail`,
   `updatedAt`). **This does not exist yet**: it must be defined and *written* by the
   existing `SessionStart`/`PreToolUse` hooks (and agent lifecycle wrappers), then
   read by the read model. The only structured agent signal today is Claude
   Code-specific (`engine/claude-sessions.ts` reads `~/.claude/jobs/<id>/state.json`);
   the marker generalizes this across agents.

Tradeoff (accepted): less real-time granularity for reliable, agent-agnostic signals.

### 4.3 Observer daemon — the new seam (to build; do NOT reuse the MCP server)

The observer daemon is the only structural addition, and it is **genuinely new**:

- **Per-repo**, addressed by a local **pipe/socket** derived from the repo path.
- **File-watcher** on the signals; **incremental** recompute.
- **Idle-timeout self-shutdown** — no always-on cost.
- **Multi-client**: serves the Neovim plugin, an optional status TUI, and external
  clients from one place.

**Critical constraint / decision.** The existing MCP server (`src/mcp/server.ts`) is
`StdioServerTransport`, spawned **per agent** by the adapters, holding **in-memory
identity**. It **cannot** be the shared, multi-client, long-lived observer daemon.
The plan must decide explicitly:

> Does `rw mcp` stay a thin per-agent stdio client that talks *to* the new daemon,
> or does the daemon expose an MCP endpoint of its own? These are two different
> transports today; picking one is a Phase-2 gate.

The daemon is a **read/notify** service over the existing core; it does not run
agents.

### 4.4 Cockpit — mode 1: STATUS dashboard

A lazygit-style, single-pane, keyboard-only view of all sessions (area, claim, git,
light), built on the Phase-1 read model. Form (standalone TUI vs a Neovim buffer) is
open (§7); a thin daemon client either way.

### 4.5 Review table — mode 2: Neovim plugin

The **mesa de revisiones** is a **Neovim plugin** (the user is a LazyVim power user),
never a rebuilt editor:

- **Sessions pane** — from claim/status signals.
- **Diff + edit buffer** — the session's `git diff` vs the integration branch in a
  **real Neovim buffer**: preview *and* native editing with the user's LSP/keymaps.
- **Evidence pane** — reads `.review-board/`. The integrator report and artifact tree
  exist; the plan adds per-session `sessions/<id>/status.json` and evidence.
- **Diagram pane** — dev diagrams generated from `.codegraph/` (§4.6).
- **Actions** — approve/reject/comment → **new decision records** in the backing
  store + release the `integrator` claim via `engine/identity.ts`.

A **Lua RPC client** of the observer daemon. Reviews and edits; does not run agents
and does not perform the merge (that is `engine/lifecycle.ts` / `rw_finish`, invoked
deliberately).

### 4.6 CodeGraph consumption (consume via its interface, not its DB)

rw-ai does **not** build a symbol index. It queries the repo's existing `.codegraph/`
through **CodeGraph's own interface** (`codegraph_explore` MCP tool / `codegraph`
CLI) — never the versioned SQLite file — for blast-radius and rendered diagrams. If
absent, diagrams degrade to ASCII.

## 5. Boundaries (non-goals)

- No terminal wrapping / PTY / pane management.
- No stdout regex parsing for agent state (structured signals only).
- No rebuilt editor (compose with real Neovim).
- No code-symbol index of our own (consume CodeGraph via its interface).
- No agent execution and no automatic merge from the review surface.
- No always-on heavy daemon (lazy + idle-timeout only).
- The MCP server stays a per-agent integration surface; it is **not** the daemon.
- Core stays agent-agnostic and token-free.

## 6. Implementation sequence

- **Phase 0 — Existing core. DONE (on `origin/main`).** Contract, provisioning
  engine, claims lifecycle, MCP tools, hooks/adapters, integrator + review-board
  substrate. (A fresh audit of this real code is the first Phase-0 task — the earlier
  audit covered the `redesign-brain` line, not this one.)
- **Phase 1 — Read model + hook-marker contract.** Define + wire the
  `sessions/<id>/status.json` marker via existing hooks; add ahead/behind to
  `engine/git.ts`; aggregate claims + git + markers into per-session light; ship
  `rw status` (the command does not exist today).
- **Phase 2 — Observer daemon.** New lazy per-repo daemon (pipe/socket, watcher,
  incremental, idle-timeout, multi-client) serving the Phase-1 model. Resolve the
  daemon-vs-MCP-transport decision (§4.3).
- **Phase 3 — Neovim plugin (STATUS + REVIEW).** Lua RPC client; sessions/diff/edit/
  evidence; decision records + integrator-claim release.
- **Phase 4 — Diagrams from CodeGraph.** Blast-radius via CodeGraph's interface,
  `image.nvim` render, ASCII fallback.
- **Phase 5 — External client (reframed).** The claims lifecycle already has one
  owner and the MCP tool surface already exists; the only open question is whether the
  **daemon** mutates claims or delegates to `engine/identity.ts`, and whether external
  AI agents/servers attach through the daemon or the existing MCP tools.

## 7. Open decisions

- **Daemon vs MCP transport** (§4.3) — the key Phase-2 gate.
- **Status dashboard form** — standalone TUI vs a Neovim buffer.
- **Daemon location** — rw-ai core vs a separate adapter package.
- **Hook-marker schema** — exact fields and the git thresholds mapping to 🟢🟡🔴,
  and which hooks/wrappers write it per agent.
- **`redesign-brain` reconciliation** — decide which ideas (external worktree
  adoption, runtime manifests) to fold into origin, if any.
- **Version drift** — `package.json` is `0.1.0` while the milestone is "v0.5".
