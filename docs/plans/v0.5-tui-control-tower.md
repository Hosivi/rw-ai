# rw-ai v0.5 — Terminal Control Tower (aligned plan)

> Status: current direction. This plan **supersedes** the ORCA-ownership stance in
> [`../architecture/orca-rwai-runtime-model.md`](../architecture/orca-rwai-runtime-model.md)
> and the startup posture in
> [`../architecture/startup-integration.md`](../architecture/startup-integration.md).
> Where those documents assume an external IDE (ORCA) owns the visual/orchestration
> surface, this plan makes rw-ai's own terminal surface that layer — while keeping
> rw-ai's core agent-agnostic and headless.

## 1. Why this plan exists (the pivot)

The original intent of rw-ai was a **review table** plus **worktree and branch
administration** for parallel agent sessions across different **areas** of a
codebase — i.e. **feature-based** agent development, not layer-based.

Two intermediate designs were evaluated and rejected against the real workflow:

| Tool | Verdict | Reason |
| --- | --- | --- |
| ORCA IDE | Rejected | Too slow, not keyboard-friendly, uncomfortable for a terminal-first flow. |
| Herdr | Rejected | Expected to be the solution; did not fit in practice. |

The decision: rw-ai provides its **own terminal-native surface** that is **100%
keyboard-driven**, fast, and composes with the tools the user already lives in
(**Zellij/Tmux + Neovim/LazyVim**) instead of replacing them.

## 2. Product identity — "Option A" (observe & compose)

rw-ai stays a **headless local control-plane** (the "brain") and gains a **thin,
read-oriented surface** (the "cockpit") that **observes** structured state. It does
**not** wrap agent terminals, manage panes, or embed an editor.

The rejected alternative ("Option B": a daemon that PTY-wraps agents, parses their
stdout for state, and embeds terminals in a full-screen TUI) is explicitly out of
scope: it rebuilds what Zellij/Tmux already do better, reintroduces the latency that
made ORCA unusable, and couples rw-ai to each agent's terminal output.

**Guiding rule: compose, don't compete.** Every time the surface is tempted to
manage panes, wrap a terminal, or rebuild an editor, that is the ORCA/Herdr trap.

## 4. Components

### 4.1 Headless brain — already built (do not touch to add UI)

The current package is the control-plane data layer and is verified (118 tests):

- **Contract** (`src/contract/schema.ts`): agent-agnostic `agents.config.json` —
  project/stacks, git config, board dir, db (no secrets), `sharedZones`, `tests`,
  and `sessions[]` with `id/branch/worktree/status/areas/ports/db/platforms`, plus
  claims (`free`/`claimed`) and roles (`sessionId | integrator`).
- **Defaults** (`src/contract/defaults.ts`): deterministic session/port/db factories.
- **Env** (`src/contract/env.ts`): per-session `.env.local` generation with a
  `${RW_DB_PASSWORD}` placeholder — secrets never land in files.
- **Runtime** (`src/runtime.ts`): builds/writes a runtime-manifest per **externally
  adopted** worktree (`mode: 'external'`), with path-containment safety.
- **Bootstrap** (`src/bootstrap.ts`): atomic writes, a bootstrap lock, and a
  pending-runtime-adoption marker state machine.

This layer is the single source of truth. Every surface reads from it; nothing
below duplicates it.

### 4.2 Structured state signals (no stdout scraping)

Agent state (🟢 free/idle · 🟡 working/attention · 🔴 blocked/failed) is derived
from **structured signals**, never by regex over an agent's terminal output:

1. **`claims.json`** — role occupancy (free / claimed / expired).
2. **git state per worktree** — dirty / staged / ahead-behind vs the base branch.
3. **hook markers** — lifecycle hooks of the agent (Claude Code, OpenCode, …) write
   small status files into `.review-board/sessions/<id>/` (e.g. `tests: passed`,
   `blocked: waiting-input`). rw-ai reads these; it does not intercept the terminal.

Tradeoff (accepted): less real-time granularity ("thinking right now") in exchange
for reliable, agent-agnostic signals ("s2 is dirty / claim held / last test failed")
— the correct granularity for a worktree + review cockpit.

### 4.3 Observer daemon — the single seam (to build)

A **lazy, per-repo** background process — the only structural addition in v0.5.
Design borrowed from CodeGraph's per-repo indexing daemon:

- **Per-repo**, addressed by a local **pipe/socket** derived from the repo path.
- **File-watcher** on the structured signals; **incremental** updates only.
- **Idle-timeout self-shutdown** — no always-on cost; it spawns on demand and dies
  when idle. This preserves the lightweight feel; a heavy always-on daemon would be
  a regression toward Option B.
- **Multi-client**: it serves *all* consumers of workspace state from one place —
  the Neovim plugin, the optional status TUI, and (future) an external AI
  agent/server. One seam, not three.

The daemon is a **read/notify** service over the headless brain and backing store.
It does not run agents and does not mutate the contract beyond what the CLI already
does.

### 4.4 Cockpit — mode 1: STATUS dashboard

A lazygit-style, single-pane, keyboard-only view of all sessions: id, area, claim
state, git state, and 🟢🟡🔴 derived from the signals above. Form (standalone TUI
vs a Neovim buffer) is an open decision (§7); it is a thin client of the daemon
either way.

### 4.5 Review table — mode 2: Neovim plugin

The **mesa de revisiones** is the origin of the project and is a **Neovim plugin**
(the user is a LazyVim power user), not a rebuilt editor:

- **Sessions pane**: which sessions need review, driven by claim/status signals.
- **Diff + edit buffer**: the session's `git diff` vs the integration branch, in a
  **real Neovim buffer** — preview *and* native editing with the user's LSP and
  keymaps. The integrator can fix code inline during review.
- **Evidence pane**: test results, screenshots, summary, risks, handoff notes read
  from `.review-board/sessions/<id>/`.
- **Diagram pane**: development diagrams (call-graph, dependency, blast-radius)
  **generated from the existing `.codegraph/` index** and rendered via `image.nvim`
  (Kitty/WezTerm/sixel), degrading to ASCII where unsupported.
- **Actions** (keyboard): approve / reject / comment → recorded as decision/risk
  records in the backing store and releases the `integrator` claim.

The review surface is a **Lua RPC client** of the observer daemon.

Boundary: the review table **reviews and edits**; it does **not** execute agents and
does **not** perform the merge (that stays with the agent or the human integrator).

### 4.6 CodeGraph consumption (consume, not rebuild)

rw-ai does **not** build a symbol index — that is a different product and would
contradict the headless-coordinator identity. Instead it **consumes** the per-repo
`.codegraph/` graph that already exists (nodes = symbols, edges = calls/contains/
imports) to enrich the review table: blast-radius of a diff and the rendered
development diagrams. If `.codegraph/` is absent, diagrams degrade gracefully.

## 5. Boundaries (non-goals)

- No terminal wrapping / PTY / pane management (Zellij/Tmux own that).
- No stdout regex parsing for agent state (structured signals only).
- No rebuilt editor (compose with the real Neovim).
- No code-symbol index of our own (consume CodeGraph).
- No agent execution and no automatic merge from the review surface.
- No always-on heavy daemon (lazy + idle-timeout only).
- Core stays agent-agnostic and token-free; no model/token management.

## 6. Implementation sequence

- **Phase 0 — Headless brain. DONE.** Contract, defaults, env, runtime-manifests,
  bootstrap (118 tests green).
- **Phase 1 — Read model + hook-marker contract.** Define the marker format written
  to `.review-board/sessions/<id>/` and a pure function that computes current
  workspace state from `claims.json` + git + markers. Ship as `rw status` (one-shot,
  no daemon yet). Fully unit-testable.
- **Phase 2 — Observer daemon.** Lazy per-repo daemon over a pipe/socket with a
  file-watcher, incremental updates, idle-timeout, and a multi-client read/notify
  protocol serving the Phase 1 read model.
- **Phase 3 — Neovim plugin (STATUS + REVIEW).** Lua RPC client to the daemon.
  Sessions pane, diff/edit buffer (native), evidence pane, and the review actions
  that write decisions back and release the integrator claim.
- **Phase 4 — Diagrams from CodeGraph.** Call-graph / blast-radius generated from
  `.codegraph/`, rendered via `image.nvim`, ASCII fallback.
- **Phase 5 (optional) — External client + MCP reconciliation.** Expose the daemon
  to an external AI agent/server as another client, and reconcile the claims
  lifecycle surface (`rw_claim`/`rw_status`/`rw_release`/…) that currently lives in a
  separate MCP server, not in this repo (see §7).

## 7. Open decisions

- **Status dashboard form**: standalone TUI vs a Neovim buffer. (Review surface is
  already decided: Neovim plugin.)
- **Daemon location**: rw-ai core vs a separate adapter package.
- **State-detection detail**: exact hook-marker schema and the git-signal thresholds
  that map to 🟢🟡🔴.
- **MCP-surface reconciliation**: the claims lifecycle contract is defined in this
  repo but implemented by an out-of-repo MCP server (`rw_claim`/`rw_release`/…). The
  CLI here only ships `rw prepare`. Decide whether the daemon owns claim mutation or
  delegates to that MCP server.
